决策点一：权限方案

四种方案需要的特权及对应的安全风险

特权对比矩阵

特权DockerBuildah
vfs		Buildah
overlay		BuildKit作用风险
Docker socket ✔——— 控制宿主机 Docker daemon 攻击者可创建特权容器实现逃逸
SYS_ADMIN —✔✔— 挂载文件系统、创建隔离命名空间 高危系统调用（mount、加载内核模块等）
Seccomp 无限制 —✔✔部分放宽 放行 mount、unshare 等系统调用 高风险调用可执行，降低逃逸难度
AppArmor 无限制 —✔✔部分放宽 支持容器内挂载 overlayfs 可访问 /proc、/sys 等特殊文件
系统路径无限制 —✔✔— 允许访问所有虚拟文件系统 可访问 /proc、/sys，降低逃逸难度
FUSE 设备 ——✔— 支持 fuse-overlayfs，提高构建速度 风险较低，需配合其他特权
BuildKit 不使用 SYS_ADMIN、不放开 systempaths，仅需 apparmor/seccomp/landlock 部分放宽（非全局 unconfined）。需配置 kernel.apparmor_restrict_unprivileged_userns=0

安全风险：逃逸路径与难度

方案核心攻击面关键逃逸路径逃逸难度评级
Docker docker.sock → 宿主机 root docker run --privileged -v /:/host 极低（一条命令） 高风险
Buildah vfs SYS_ADMIN + seccomp/apparmor 关闭 cgroup release_agent、mount 宿主机磁盘设备 中等 中高风险
Buildah overlay vfs 攻击面 + /dev/fuse 上述全部 + FUSE 文件系统劫持、overlay 镜像层投毒 中低（最多路径） 高风险
BuildKit seccomp/apparmor 部分放宽（无 SYS_ADMIN） 待穿刺确认（攻击面显著小于 Buildah） 待评估（预期高） 低风险

决策点二：部署方式

镜像处理模块的进程模型选择

合并部署 vs 独立部署

维度合并部署（进程内）独立部署
架构 镜像处理模块作为控制面进程内调用 镜像处理模块作为独立进程运行
资源隔离 构建镜像与控制面共享资源 构建资源独立分配，不影响 API 响应
部署复杂度 无需额外组件 需启动独立特权容器
故障隔离 构建崩溃可能影响 API 服务 构建崩溃不影响 API 服务
适用阶段 MVP / 功能验证 生产环境
缓解措施：构建容器通过 Unix socket 或 HTTPS 暴露接口 + 非对称加密密钥或配置文件权限访问控制；严格校验构建参数，禁止直接运行上传的二进制。

当前推荐

决策维度当前选择理由
构建方案 Buildah + overlay Daemonless、性能中等、已有实测数据；通过 seccomp profile 精细控制可降风险
部署方式 合并部署 MVP 阶段快速验证，部署简单；后续生产化时切换为独立部署
加固措施 自定义 seccomp profile（移除 kexec、bpf 等危险调用）+ AppArmor profile + 构建接口输入校验
演进方向 BuildKit 穿刺完成后评估切换（原生无需 SYS_ADMIN，安全评级显著更优）

总结

关键结论 权限：Docker socket 和 SYS_ADMIN 是最大攻击面，BuildKit 是长远安全最优解
部署：合并部署降低复杂度，独立部署提升安全隔离——按阶段演进
当前方案Buildah overlay + 合并部署 + seccomp/AppArmor 精细加固定
下一步① 编写自定义 seccomp/AppArmor profile ② 完成 BuildKit 穿刺 ③ 生产化时切换到独立部署
